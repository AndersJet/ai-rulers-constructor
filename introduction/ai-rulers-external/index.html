大家好，今天给大家介绍一个开源项目——Rulers Template。

一句话概括：它是给 AI 编程助手装上的"交通护栏"。如果你在用 Cursor、Copilot 或 Claude Code 写代码，这个项目能让 AI 的输出更安全、更一致、更符合你的项目规范。

我自己在用 AI 写代码的过程中踩过很多坑——AI 擅自改认证逻辑、每次输出的代码风格不一样、同一个需求我和同事的 AI 给出了完全不同的方案。这些问题的根源不是 AI 不够聪明，而是它缺少边界。

今天大概花 15 到 20 分钟，先聊痛点，再讲方案，最后告诉大家怎么三分钟上手。有问题随时打断。

// part 01 · 痛点

你遇到过这些问题吗？

🎲 风格随机

AI 这次写的代码风格，和上次完全不同——命名、缩进、错误处理全都不一样。

🔓 越权操作

AI 擅自改了认证逻辑，你 review 时吓出一身冷汗——它甚至没提醒你这是高风险改动。

🗄️ 数据风险

AI 直接删了数据库字段，没提醒你备份，也没给 rollback 脚本。

🔄 输出不一致

同一个需求，你和同事的 AI 给出了完全不同的实现方案——规范在哪？

我相信用 AI 写代码的朋友都遇到过这四种情况。

第一个，风格随机。同一个项目，今天和明天生成的代码风格完全不同——命名方式变了、缩进变了、错误处理模式变了。每次 code review 都在修风格问题。

第二个，越权操作。AI 擅自改了认证逻辑，review 的时候吓出一身冷汗——它甚至不提醒你"兄弟，这是高风险改动"。

第三个，数据风险。AI 直接生成 DROP COLUMN 语句，不提醒备份，不给 rollback。如果没仔细看就直接执行了，后果真的不堪设想。

第四个，输出不一致。你和同事都给 AI 同一个需求，结果两个人拿到了完全不同的实现方案——一个用 ORM，一个写原生 SQL——团队规范形同虚设。

这些问题的根源是什么？不是 AI 不够聪明——而是它缺少边界。下一页展开讲。

// part 01 · 根源

AI 越强大，边界越重要

AI 编码助手本质上是通用工具——它不了解你项目的特定约束。

①
上下文窗口有限

AI 一次只能"看到"一小部分代码，不了解你的项目全貌和架构决策。

②
会话独立无记忆

每次新会话都是"重新认识"你的项目，上次的约定全部丢失。

③
通用训练数据

训练数据来自互联网，不包含你项目的编码规范、技术选型和业务约束。

④
无安全感知

AI 不知道哪些操作是"禁区"——它不会区分开发环境和生产环境的风险。

AI 编码助手本质上是通用工具，它不了解你项目的特定约束。四个原因。

第一，上下文窗口有限。AI 一次只能看一小部分代码，不了解项目全貌。你的架构决策、技术债务、历史原因——它都不知道。

第二，会话完全独立。每次开新会话，AI 就是"重新认识"你的项目。上一次你花半小时教它的约定，全没了。

第三，通用训练数据。AI 学的是互联网上的代码，不包含你项目的编码规范。它不知道你们团队决定用 class-validator 而不是 Joi，不知道你们 API 返回格式统一是 {code, data, message}。

第四，没有安全感知。AI 分不清开发环境还是生产环境。对它来说，改一个 CSS 颜色和删一个数据库字段——都是"修改文件"。

所以我们需要的方案很明确：给 AI 一套项目专属的规则手册——不是限制它，而是保护它不犯错。

Rulers Template 就是阶段 C 的开源实现。

行业在 AI 编码规范这件事上，大致经历了三个阶段。

阶段 A——无规则 vibe coding。一开始大家都没规则，靠个人提示词能力。很快，但质量完全不可控。一个人能驾驭，一个团队就乱了。

阶段 B——大 AGENTS.md 单文件。有人开始写一个巨大的 AGENTS.md，所有规则全塞进去。有了规范，但文件越来越长，重要规则被噪声淹没。改一处可能影响全局。

阶段 C——目录索引 + 渐进式加载。这是当前的最佳实践：把规则按领域拆成目录结构，AI 只加载当前任务需要的规则。后端任务不加载前端规则，数据库 migration 不加载 CSS 规范。

Rulers Template 就是阶段 C 的开源实现。它提供了一套完整的模板和自动化工具，让你几分钟内就能从阶段 A 跨到阶段 C。

Rulers Template 的工作方式用一张流程图就能讲清楚。

你的项目代码库是起点——真实的文件、配置、目录结构。AI 对你的项目做九维度自动扫描——技术栈、安全模型、数据库、CI/CD、命令等等。

扫描完成后生成PROJECT_PROFILE.md——这是一份项目画像，记录已确认的事实："这个项目用 Express + TypeORM"、"数据库是 PostgreSQL"、"CI 用 GitHub Actions"。

然后基于画像，为每个领域生成专属规则——backend 有 API 设计和数据访问规范，database 有 migration 和备份规范，frontend 有组件和样式规范。

最后一步是人工审阅激活——不是生成完就全生效，而是你逐级审阅、逐级解锁。这个安全机制下一张片子细讲。

最重要的是：初始化完成之后，每次 AI 会话自动遵循这些规则。你不需要再说"注意安全""注意风格"——规则帮你说了。

// part 02 · 设计原则

不是"更多规则"，
而是更聪明的规则

🔍 证据驱动

规则基于项目真实的文件、配置和命令生成——绝不编造。每条规则都有据可查。

⚡ 渐进加载

AI 只加载当前任务相关的规则，不浪费宝贵的上下文窗口。后端任务不加载前端规则。

🔒 激活门禁

四级安全等级，规则未经人工审阅不能执行危险操作。给 AI 装上权限系统。

🧭 领域路由

后端、数据库、前端——每个领域有独立的约束集，互不干扰，各管各的。

四个核心设计理念，每个都围绕一个关键问题。

证据驱动——你肯定见过那种"AI 编码规范大全"的博客，写了几十条规则但跟你的项目没关系。Rulers 不一样——规则基于你项目的真实代码生成，绝不编造。文件说用 Express，规则就写 Express；配置里是 PostgreSQL，规则就写 PostgreSQL。

渐进加载——AI 不会一口气加载所有规则。上下文窗口是稀缺资源，我们只加载当前任务需要的。做后端接口只加载 backend 规则，做数据库 migration 只加载 database 规则。

激活门禁——这个最重要。不是生成完就全放开，而是有四级安全等级。高风险操作必须人工审阅后才能执行。

领域路由——后端、数据库、前端各有各的约束，互不干扰。每个领域有独立的 INDEX.md 作为索引。

这四个原则加起来，让 Rulers 不是一个"规则大全"，而是一个规则系统。

// part 02 · 安全兜底

给 AI 装上权限系统

四级激活门禁——不是一开始就全放开

L0
模板已部署

仅项目发现——AI 只能看，不能动。你需要确认项目目录名。

L1
项目画像已审阅

允许只读分析、低风险文档、小型本地修复。你需要审阅 PROJECT_PROFILE.md。

L2
领域规则已审阅

允许在已激活领域内开发。认证/安全变更、数据库 migration 至少需此级。

L3
发布门禁已审阅

允许有生产影响的工作。CI/CD 变更必须到此级。

四级激活门禁是 Rulers 最重要的安全机制。一句话概括：不是生成完就全放开，而是逐级解锁。

Level 0——模板刚放进来。AI 只能做项目发现，不能改任何代码。你想让它写代码？不行，先升级。

Level 1——项目画像已审阅。你看过了 PROJECT_PROFILE.md，确认里面的技术栈、命令、项目事实都是对的。这时候 AI 可以做简单的只读分析和文档工作了。

Level 2——领域规则已审阅。这是大部分日常开发所在的等级。你审阅了对应领域的规则，AI 可以在那个领域内开发。注意：认证变更、数据库 migration 这些高风险操作，最低要求就是 Level 2。

Level 3——发布门禁已审阅。CI/CD 部署相关的变更必须到这个等级。这是生产环境的最后一道防线。

这个设计相当于给 AI 装了一个权限系统——不是什么都能碰，什么操作需要什么等级，明明白白。

// part 03 · 上手

从零到可用，只需一句话

# 1. 安装 skill（一次性，30 秒） cp -r ai-rulers-template/skills/ai-rulers-init/ ~/.claude/skills/ # 2. 在项目中，对 AI 说出那句魔法指令： "使用 ai-rulers-init skill，帮我初始化 AI rulers" # AI 自动完成： # 扫描项目 → 生成画像 → 生成规则 → 校验 → 你审阅激活 # ← 全自动，几分钟内完成 →

两种方式任选：Skill 一键初始化，或手动拷贝模板。

听起来很复杂？其实上手只需要两件事。

第一步，安装 skill——一条 cp 命令，30 秒。一次安装，所有项目通用。

第二步，打开你的项目，对 AI 说一句："使用 ai-rulers-init skill，帮我初始化 AI rulers"。

然后 AI 自动完成一切：扫描项目技术栈、生成项目画像、为每个领域生成规则、跑校验脚本检查完整性。你只需要做一件事——审阅并确认激活。

整个过程全自动，通常几分钟内完成。不是"下载下来再手动改半天"——是真正的一键初始化。

如果你不想装 skill，也可以用传统方式：手动拷贝模板目录，然后对 AI 说"加载 AGENTS.md，按初始化流程执行"。两种方式效果一样。

// part 03 · 日常

初始化之后——无感运行

你对 AI 说："帮我加一个用户列表接口" AI 自动加载规则链： AGENTS.md → core/* → PROJECT_PROFILE.md → backend/INDEX.md → api-design.md AI 输出自动遵循： ✓ 命名规范（Controller → Service → Repository） ✓ 分页参数命名用 page / page_size ✓ 返回格式统一 { code, data, pagination } ✓ 输入校验用 class-validator ✓ 错误处理用统一异常过滤器

初始化完成之后，日常使用是完全无感的。你不需要每次跟 AI 说"注意风格""注意安全"——规则自动生效。

举个实际例子。你对 AI 说"帮我加一个用户列表接口"。AI 自动加载规则链：从 AGENTS.md 入口开始，到 core 全局约束，到 PROJECT_PROFILE 项目画像——知道了你用 Express + TypeORM——再到 backend 领域索引，最后到具体的 api-design 规则。

输出的代码自动遵循你们项目的所有约定：分层架构是 Controller-Service-Repository、分页参数叫 page 和 page_size、返回格式统一是 {code, data, pagination}、输入校验用 class-validator——全部自动对齐。

这就是"护栏"的真正含义——不是每次都手动提醒 AI，而是规则自动帮你说。

这个方向不是我们在自说自话。学术界和工业界都在关注。

AGENTS.md 已经成为一个跨工具的社区标准，强调可预测位置、就近覆盖、可持续维护。

有一篇今年发表在 arXiv 上的研究（2601.20404），专门分析了 AGENTS.md 的影响——他们发现在样本中，AGENTS.md 的使用与运行时长和 token 消耗下降相关。换句话说，有了规则，AI 反而更高效。

另一篇研究（2511.12884）分析了很多 Agent READMEs 文件，发现一个关键问题——安全约束和性能约束的覆盖率明显偏低。这正是 Rulers Template 专门解决的：它有专门的 HARD_CONSTRAINTS.md 和安全规则。

Claude Code 官方最佳实践也强调：上下文窗口是稀缺资源，规则应该模块化拆分，不要塞进一个大文件——这恰好是 Rulers 的"渐进式加载"理念。

所以我们不是拍脑袋做的，是基于行业趋势和学术证据。

  • AI 扫描项目自动生成
  • 内置 4 大领域 + 可扩展
  • 四级激活门禁
  • 自动校验脚本
  • Claude Code / Cursor / Copilot

和其他方案对比一下，让大家看清楚差异。

手写 AGENTS.md——很多人这么做。问题是高度依赖个人经验，你得知道该写什么、怎么组织。而且随着项目演进，维护成本越来越高。没有安全等级、没有自动校验。

Cursor Rules——只支持 Cursor 一个工具。单文件模式，项目大了之后和阶段 B 的大 AGENTS.md 是同一个问题。也没有安全等级和校验能力。

Rulers Template 的差异化很清晰：AI 自动扫描项目生成——不用你自己想该写什么；内置四大领域加可扩展——结构是现成的；四级激活门禁——安全兜底；自动校验脚本——改了规则跑一下就知道对不对；跨工具通用——Claude Code、Cursor、Copilot 都支持。

简单说，手写方案是"我想好规则再写下来"，Rulers 是"AI 帮我发现规则，我审阅确认"。
