// part 01 · 认知对齐

AI 写代码的三个随机性

🎲 行为随机

同样需求，两次输出可能完全不同。风格不一致、模式不统一——每次 code review 都在修命名和格式。

🔇 知识断层

AI 不了解你项目的技术选型和约定。它不知道你用 class-validator 还是 Joi，不知道你 API 返回格式是 {code, data}。

🚫 安全意识为零

AI 不知道哪些操作是危险区。对它来说，改 CSS 颜色和删数据库字段——都是"修改文件"。

先快速对齐认知——为什么 AI 编码必须要有规范。不是因为"规范很重要"这种正确的废话，而是 AI 有三个天生的缺陷。

第一个，行为随机。同样的需求，你问两次，AI 给两次答案——可能完全不一样。变量命名、文件结构、错误处理方式——每次都可能不同。这就是为什么 code review 里大量时间花在修风格问题上。

第二个，知识断层。AI 是通用模型，不了解你项目的具体约定。你们团队决定 API 返回格式统一用 {code, data, message}，但 AI 不知道——它可能返回 {status, result}。你每次都要手动纠正，这就是认知负担。

第三个，安全意识为零。这一点最要命。对 AI 来说，修改一个 CSS 颜色和删除一个数据库字段没有区别——都是"修改文件"。它不会提醒你"这个操作需要备份""这个改动影响生产环境"。

这三个问题的共同根源是什么？AI 缺少关于你项目的上下文约束。这就是规范体系要解决的问题。

Rulers Constructor = 阶段 C 的开源实践。接下来我们看怎么用。

行业在这件事上经历了三个阶段，了解这个演进路径有助于理解"为什么 Rulers 是这个结构"。

阶段 A——无规则，也就是所谓的 vibe coding。靠个人提示词能力驱动 AI。短期内很快，一个人的时候没问题——但团队协作就崩了，质量全靠运气。

阶段 B——大 AGENTS.md。有人意识到需要规范，于是写了一个巨大的文件，所有规则全塞在里面。刚开始有用，但随着规则越来越多，文件越来越长——重要规则被噪声淹没，改一处影响全局，维护成本飙升。

我们团队有些人可能经历过前两个阶段——知道痛点在哪。

阶段 C 是当前的最佳实践：目录索引加渐进式加载。规则按领域拆分成独立文件，AI 只加载当前任务相关的。后端任务不碰前端规则，数据库 migration 不加载 CSS 规范。

Rulers Constructor 把这个模式做成了开箱即用的模板——你不用自己设计目录结构、不用自己写索引机制、不用自己搭校验——全都现成的。

// part 02 · 结构

一张地图：模板里有什么

documents/rulers/ ├── AGENTS.md ← 🔴 AI 入口协议（初始化时脚本会拷到项目根目录） ├── PROJECT_PROFILE.template.md ← 项目画像模板 ├── core/ ← 🔴 全局常驻规则（5个文件，每次会话始终加载） │ ├── HARD_CONSTRAINTS.md ← 全局硬约束——安全底线，优先级最高 │ ├── WORKFLOW.md ← 推理与执行流程 │ ├── DOC_GOVERNANCE.md ← 规范文档自身的治理规则 │ ├── RULER_MAINTENANCE.md ← 规则维护决策树 │ └── GIT_COMMIT_CONVENTION.md ← Git 提交规范 ├── backend/ ← 🟡 后端领域（按需加载） ├── database/ ← 🟡 数据库领域 ├── frontend/ ← 🟡 前端领域（含 common/web/app 子路径） ├── bootstrap/ ← 🟢 引导与生成指南 └── scripts/ ← 🟢 validate_rulers.py 校验脚本

这张地图大家先留个印象，后面会反复用到。记住三色分层。

红色——core 目录。5 个文件，每次 AI 会话都会加载，是全局底线。最重要的是 HARD_CONSTRAINTS.md——安全硬约束，比如"不能删除生产数据库""不能修改认证中间件的签名验证逻辑"。还有 WORKFLOW.md——控制 AI 的推理和执行流程。

黄色——领域目录。backend、database、frontend——按需加载。做后端任务只加载 backend，做 migration 只加载 database。每个领域内部有一个 INDEX.md 作为索引，AI 通过索引找到具体规则文件。

绿色——辅助目录。bootstrap 里面有初始化流程指南和规则生成指南，scripts 里面有自动校验脚本——初始化时用，日常也用。

一个关键点：AGENTS.md 在模板里，初始化时会自动复制到项目根目录。这是 AI 读取的入口。后面讲初始化流程时会演示。

所有东西都有清晰的归属——这就是"结构化"的含义。

// part 02 · 加载机制

AI 怎么知道该加载哪些规则？

关键词触发 → 索引路由 → 按需装配

你的任务："帮我写一个用户注册接口" AI 解析关键词 → 匹配 trigger_keywords → 加载链路： AGENTS.md （入口协议） → core/HARD_CONSTRAINTS.md （安全底线——始终加载） → core/WORKFLOW.md （执行流程——始终加载） → PROJECT_PROFILE.md （项目事实：Express + TypeORM + PostgreSQL） → backend/INDEX.md （后端规则索引） → backend/api-design.md （API 设计规范） → backend/api-security.md （安全规范） → backend/data-access.md （数据访问层规范） # trigger_keywords 采用中英双语，确保跨模型命中率： # - api, endpoint, 接口, API, 路由

大家可能会好奇：AI 怎么知道该加载哪个规则？答案是关键词触发 + 索引路由。

每个规则文件都有一个 metadata 区域，其中包含 trigger_keywords。当你对 AI 说"帮我写一个用户注册接口"，AI 会解析出"接口""API"这些关键词，然后匹配到 backend 领域的规则。

注意一个细节：trigger_keywords 是中英双语的。同一个规则文件可能同时包含 "api""endpoint""接口""API""路由"——这样不管你说中文还是英文，都能命中。这是因为不同 AI 模型对中英文的敏感度不同，双语覆盖确保跨模型命中率。

加载链路有明确的顺序：AGENTS.md 入口 → core 全局约束（始终加载）→ PROJECT_PROFILE 项目画像 → 领域 INDEX → 具体叶子规则。不是加载全部，只加载相关的——这就是"渐进式"的含义。

理解这个机制很重要，因为后面的实战都会用到——你描述任务时带的领域关键词越准确，AI 的规则匹配就越精准。

// part 02 · 安全机制

不是装上了就全生效——
四级激活门禁

L0
模板已部署

AI 只允许做项目发现——只能看，不能动。你需要确认目录名。

L1
项目画像已审阅

允许只读分析、文档写作、小型本地修复。你要审阅 PROJECT_PROFILE.md。

L2
领域规则已审阅

允许在已激活领域内开发。认证/安全变更、数据库 migration 至少需此级。

L3
发布门禁已审阅

允许有生产影响的工作。CI/CD 变更必须到此级。

核心原则：越危险的操作，需要的等级越高。

激活等级是安全兜底机制，也是 Rulers 区别于手写 AGENTS.md 的一个关键设计。

刚初始化完是 Level 0——模板已经在你项目里了，但 AI 只能看不能动。它只能做"项目发现"——扫描你的代码库，生成画像。你想让它写代码？不行，先升级。

Level 1 需要你审阅 PROJECT_PROFILE.md——确认里面的技术栈、命令、CI/CD 配置都是对的。审阅完，AI 可以做只读分析和小修复了。

Level 2 是日常开发的主要等级。你需要逐领域审阅规则——比如先审阅 backend 的规则，审完 AI 就能在后端领域开发。注意：认证变更、数据库 migration 这种高风险操作，最低要求就是 Level 2。

Level 3 是最高级——CI/CD 部署相关的变更必须到这里。

关键要记住：你不是一次审完所有规则，而是用哪个领域激活哪个领域。渐进式的——审一个用一个，不审的保持 Level 0。

这保证了你在完全了解规则之前，AI 不会越权操作。相当于给 AI 装了一个权限系统。

// part 02 · 边界情况

规则冲突时谁说了算？

优先级金字塔 ┌──────────────────┐ │ core 全局硬约束 │ ← 最高 ├──────────────────┤ │ 安全规则 │ ├──────────────────┤ │ 激活门禁 │ ├──────────────────┤ │ 领域规则 │ ├──────────────────┤ │ 专题规则 │ ├──────────────────┤ │ 示例 │ ← 最低 └──────────────────┘

记住一句话

安全 > 惯例

举例

后端规则说"可以用这种模式"，但 core/HARD_CONSTRAINTS.md 说"不能调用外部未授权 API"→ core 胜出。

有时候不同规则之间可能产生冲突——比如后端规则说"可以这样做"，但全局硬约束说"不行"。谁说了算？

裁决顺序就是一个优先级金字塔。

最高优先级永远是 core 全局硬约束——这是安全底线，没有任何规则可以覆盖它。比如 HARD_CONSTRAINTS.md 里如果写了"禁止调用未授权的外部 API"，那所有领域规则都不能违反。

第二层是安全规则——认证、授权、数据保护相关的约束。

第三层是激活门禁——如果某个领域还没激活（还在 Level 0），那这个领域的任何规则都不能被"绕过"去执行。

再往下是领域规则和专题规则。最低优先级是示例——示例只是参考，不具约束力。

一句话记住：安全大于惯例。如果两个规则冲突，选更安全的那一个。

整个初始化流程七步。前三步是准备——AI 自动完成，你基本不用管。

核心从第四步开始：项目发现。AI 对你的项目做九维扫描——仓库形态（monorepo 还是单仓库）、技术栈（语言、框架、版本）、命令（build、test、lint）、安全模型、持久化方案、API 契约、前端平台、CI/CD 配置、高风险区域。这一步 AI 生成发现报告，需要你确认——因为有些信息 AI 可能判断不准。

第五步，基于你确认的发现，AI 生成 PROJECT_PROFILE.md——项目画像。它记录了项目的已确认事实：用什么语言、什么框架、数据库是什么、部署到哪里。

第六步，生成各领域规则——这一步 AI 自动完成。它会基于项目画像，为实际存在的领域生成 INDEX.md 和叶子规则文件。如果项目没有移动端，就不会生成 frontend/app/ 的规则。

第七步，校验加激活。AI 自动跑 validate_rulers.py 检查结构完整性，然后你逐级审阅激活。

整个过程大部分是自动的，你只需要在关键节点——发现结果、画像、激活——做确认。

// part 03 · 实战 1

场景一：后端 API 开发

你对 AI 说："帮我实现用户列表查询接口，带分页和搜索" AI 自动加载： PROJECT_PROFILE.md → 知道用 Express + TypeORM + PostgreSQL backend/INDEX.md → 加载 api-design.md, data-access.md, api-security.md core/WORKFLOW.md → 遵循探索-计划-实现-验证执行流程 AI 输出自动遵循： ✓ Controller → Service → Repository 分层架构 ✓ 分页参数命名用 page / page_size ✓ 搜索用 LIKE 模糊匹配，参数做 SQL 注入防护 ✓ 返回格式统一 { code, data, pagination } ✓ 输入校验用 class-validator ✓ 错误处理用统一异常过滤器

来看第一个实战场景——后端开发，这是最高频的使用场景。

你对 AI 说"帮我实现用户列表查询接口，带分页和搜索"。在没有规则的情况下，AI 可能返回任何东西——可能不用你的 ORM、可能分页参数叫 offset/limit 而不是 page/page_size、可能返回格式乱七八糟。

在有 Rulers 的情况下，AI 自动加载 PROJECT_PROFILE.md——它知道了你用 Express + TypeORM + PostgreSQL。然后加载 backend 领域规则——它知道了分层架构、命名规范、返回格式、校验方式。

输出的代码自动对齐所有规范：Controller-Service-Repository 三层、分页参数统一用 page 和 page_size、搜索参数做了 SQL 注入防护、返回格式是 {code, data, pagination}、输入用 class-validator 校验——全部自动。

你不用每次重复"记得用 class-validator""记得返回格式是 {code, data}"——规则帮你说了。这就是规范体系在日常开发中的实际价值。

// part 03 · 实战 2

场景二：数据库变更——高风险操作

❌ 没有规则时

  • AI 直接生成 DROP COLUMN 语句
  • 不提醒备份数据
  • 不要求写 rollback 脚本
  • 不检查是否影响生产环境
  • 直接修改 schema，无安全门禁

✅ 有规则时

  • AI 先检查：当前激活等级 ≥ Level 2？
  • 自动提示：请先确认备份策略
  • 要求提供 migration 文件 + rollback
  • 提醒数据迁移的影响范围
  • 输出符合项目 migration 命名规范

数据库变更是风险最高的场景之一——所以专门拿出来讲。

左边是没有规则约束时的情况：AI 可能直接给你一条 DROP COLUMN 语句。不提醒备份、不要求 rollback、不检查影响范围。如果你没仔细看就直接执行了——生产环境可能就挂了。

右边是有了 Rulers 数据库规则之后。首先 AI 会检查激活等级——数据库 migration 最低要求 Level 2。如果不满足，AI 会拒绝执行并告诉你"请先升级到 Level 2"。满足等级后，AI 会自动走安全检查清单：提示备份、要求提供 migration 文件和 rollback 脚本、提醒影响范围。

关键是——这些安全检查不是每次手动提醒 AI 的，是规则自动执行的。database 领域的规则文件里写死了这些流程，AI 加载后自动遵循。

这就是"护栏"在实践中的含义——把安全流程从"人肉记住"变成"系统强制"。

// part 03 · 实战 3

场景三：前端页面开发

"帮我做一个用户设置页面" 前端规则路由（自动判断平台）： frontend/common/ ← 设计 token、无障碍基线（始终加载） frontend/web/develop/ ← Web 端架构、状态管理、API 集成 frontend/web/design/ ← Web 端视觉系统、组件模式 如果是移动端（React Native / Flutter）： frontend/app/develop/ ← 移动端架构、UI 模式、测试 frontend/app/design/ ← 移动端 token、布局、组件、交互 # 关键：common/ 是两个平台共享的——设计 token、无障碍要求 # 平台特定的放在 web/ 或 app/ 下，按需加载

前端稍微复杂一些——因为有 Web 和移动端两条线。但 Rulers 的规则路由会自动处理。

当你做 Web 页面时，AI 只加载 frontend/common/ 和 frontend/web/。做移动端（React Native 或 Flutter）时，只加载 common 和 app。通用部分——设计 token、无障碍基线——放在 common 里，两个平台共享，不会重复写。

假设你的项目只有 Web 端没有移动端——初始化时 AI 会自动跳过 app 目录，不生成相关规则。这就是"证据驱动"——有什么平台生成什么规则。

如果你的项目后来加了移动端——重跑 skill，AI 自动检测到新平台，补充生成 app 规则，不会影响已有的 Web 规则。增量更新，不是全部重来。

这就是领域路由的价值：不管你团队是只做 Web，还是 Web 加移动端都做，规则都能精准匹配。

// part 03 · 实战 4

场景四：跨域联动——
最容易被忽视的风险

🗄️

Database

Schema 变更
Migration + Rollback
数据备份确认

⚙️

Backend

API 权限更新
新增字段校验
角色判断逻辑

🎨

Frontend

表单字段新增
权限 UI 展示
菜单显隐控制

⚠️ 权限/角色变更 → 必须做数据库 + 后端权限 + 前端 UI 三方一致性校验

第四个场景是全栈联动——这是最容易被忽视但风险最高的场景。

举个例子：给用户表加一个 is_admin 字段。这看起来是个小改动，但实际上涉及三个领域：

数据库——要写 migration 加字段，设默认值。后端——API 要更新权限检查逻辑，确保非管理员不能通过 API 改自己的 is_admin。前端——页面要根据 is_admin 显示或隐藏管理菜单，但前端显示只是 UI 层面的，真正的权限校验必须在后端。

在没有规则的情况下，很容易出现"改了后端忘了数据库，改了数据库忘了前端校验"的情况。三个人各改各的，上线后才发现：普通用户通过改 API 参数把自己变成了管理员。

Rulers 的跨域加载机制保证了三组规则同时被命中——因为 is_admin 这个变更同时触发了 database、backend、frontend 三个领域的关键词。而且 security checklist 强制要求权限变更必须做三方一致性校验。

跨域任务是检验规则体系是否完善的试金石——如果一个规范体系只在单领域内工作，跨域就崩，那是不合格的。

// part 03 · 质量保障

自动校验：validate_rulers.py

$ python3 documents/rulers/scripts/validate_rulers.py ✓ Markdown 链接有效性检查通过 ✓ metadata 完整性检查通过（applies_to / trigger_keywords / must_load_with） ✓ must_load_with 路径可解析且在模板范围内 ✓ INDEX.md 覆盖同级所有叶子文档 ✓ AGENTS.md 引用全部 5 个 core 规则 ✓ ACTIVATION_LEVELS.md 包含 Level 0-3 定义 ✓ 高风险文档包含正确的激活等级语言 所有检查通过 ✅

养成习惯：每次改完规则，跑一下校验。

校验脚本是自动化的质量保障——它在初始化完成后自动运行，日常也可以手动运行。

它检查七类问题：

Markdown 链接是否有效——有没有指向不存在文件的死链。metadata 是否完整——每个规则文件都有 applies_to、trigger_keywords、must_load_with 三个必填字段。must_load_with 引用的路径能不能解析。INDEX.md 是否覆盖了同级目录下的所有叶子文档——有没有"隐形孤儿文件"（文件存在但索引没引用，AI 永远不会加载）。AGENTS.md 是否引用了全部 5 个 core 规则。ACTIVATION_LEVELS.md 有没有完整的四级定义。高风险文档有没有标注正确的激活等级。

这七项检查覆盖了最常见的规则结构问题。最重要的是——它给你即时反馈。你改完规则跑一下，立刻知道有没有破坏结构。

养成习惯：改规则必校验——和写代码跑测试一样。这 30 秒的校验能省掉后面几小时的排查。

// part 04 · 效率技巧

用好 Rulers 的三个习惯

01
描述任务带上领域关键词

"帮我写个接口" → "帮我写个后端 API 接口，包含参数校验和错误处理"。关键词越多，命中越精准。

02
留意 AI 加载了哪些规则

每次 AI 回复开头会声明加载的规则文件列表。如果发现少加载了，直接在提示里补充领域关键词。

03
改规则 → 更新索引 → 跑校验 → 提交

这是一条铁律。改了规则不校验，就像改了代码不跑测试——能过是运气，不过才是常态。

三个日常习惯，很简单但能避免 80% 的问题。

习惯一：描述任务时带上领域关键词。不要只说"帮我写个接口"——太模糊了，AI 可能匹配不到正确的规则。说"帮我写个后端 API 接口"——"后端""API""接口"三个关键词同时命中，精准度大幅提升。一句话里包含的关键词越多，规则匹配越准确。

习惯二：留意 AI 回复时声明的规则加载列表。AI 在回复开头会列出来"本次加载了哪些规则文件"。瞄一眼——如果发现该加载的没加载，直接在下一句话里补关键词。比如发现 database 规则没加载，就说"这个改动涉及数据库 schema，请参考数据库规则"。

习惯三：改规则必校验。这是铁律——和写代码跑测试一样。改完规则 → 更新索引 → 跑 validate_rulers.py → 提交。养成肌肉记忆。

这三个习惯用好了，你日常使用 Rulers 几乎不会遇到问题。

// part 04 · 踩坑 ①

常见错误：只加文件，不更新索引

❌ 错误做法

# 你在 backend/ 下新增了一个 rule.md # 但忘记在 backend/INDEX.md 里添加引用 backend/ ├── INDEX.md ← 没有引用新文件 ├── api-design.md └── my-new-rule.md ← 新加的，但 INDEX 没更新 → AI 永远不会加载 my-new-rule.md → 你写的规则等于白写

✅ 正确做法

  1. 新增规则文件
  2. 立即更新对应 INDEX.md
  3. 运行 validate_rulers.py
  4. 校验脚本会报 "INDEX 未覆盖 XXX"
  5. 如果没有报错 → 确认索引已更新

第一个常见错误——也是最容易犯的。

你在 backend 目录下新建了一个规则文件，写了一堆很棒的约束。但是你忘了在 backend/INDEX.md 里加一行引用。结果：AI 通过 INDEX.md 找规则文件——找不到你的新文件——你的规则永远不会被加载。

AI 不会自动扫描目录发现新文件——它只通过 INDEX.md 路由。这是设计如此——因为索引提供了明确的加载顺序和条件。但这意味着你必须手动更新索引。

正确流程：新增文件 → 更新 INDEX → 跑校验。校验脚本会检查"INDEX 是否覆盖了同级所有叶子文档"——如果漏了，它会报错告诉你。所以跑校验是最后的安全网。

一句话：写规则不写索引 = 写了个没人看得到的东西。

// part 04 · 踩坑 ②

常见错误：一条规则写进多个文件

❌ 错误做法

在 backend/api-design.md 写了 API 错误码规范
又在 core/HARD_CONSTRAINTS.md 写了同样的内容
→ 两个文件有矛盾版本
→ 改一处忘了改另一处
→ 冲突发生了，但没人知道

✅ 正确做法

遵循"单一来源"原则：
全局约束 → core/
领域约束 → backend/ / database/ / frontend/
专题约束 → 对应 topic 文件
出现矛盾时 → core 安全约束 > 领域规则

第二个常见错误——同一个约束写在多个文件里，导致"一个规则，两个版本"。

比如你在 backend/api-design.md 写了一套错误码规范，过了一段时间忘了，又在 core/HARD_CONSTRAINTS.md 写了一遍——但细节不一样。后来你改了其中一处，却忘了另一处也存在。两个版本共存，冲突发生在运行时——AI 不知道用哪个。

这个问题的根因是没有遵循"单一来源"原则。

正确的做法是：每个约束只属于一个地方。全局性的、安全相关的放 core；领域特定的放对应领域目录；专题性的放单独的 topic 文件。不确定放哪里时——先在团队群里问一句，不要自己拍脑袋。

如果真的出现重复——冲突裁决优先级已经定义好了：core 硬约束 > 安全规则 > 领域规则。安全相关的永远以 core 为准。

记住：一个事实，一个来源。

// part 04 · 踩坑 ③

常见错误：跨域任务漏验证

⚠️ 真实案例 — 权限变更的典型翻车路径

后端同事改了 API → 对用户返回 is_admin 字段
前端同事改了页面 → 根据 is_admin 显示管理菜单
但是 → 没人检查数据库 migration 有没有加默认值
也没人检查 → 非管理员用户能否通过 API 直接改自己的 is_admin

✅ 正确做法：权限/角色变更 → 必须做三端校验

🗄️

Database
默认值 + NOT NULL

⚙️

Backend
权限中间件校验

🎨

Frontend
UI 显隐 + 接口拦截

第三个错误最危险——跨域任务改了部分领域，漏了其他领域的验证。

看这个真实案例：用户提了个 Issue "给用户加一个 is_admin 字段"。后端同事很快搞定了 API——对用户返回 is_admin 字段。前端同事也很高效——根据 is_admin 显示管理菜单。

看起来都搞定了，对吧？错了。两个关键问题没人发现：第一，数据库 migration 有没有给 is_admin 加默认值（默认 false，不允许 NULL）？如果没有，新用户的这个字段就是 NULL，代码里 `if (user.is_admin)` 可能永远走不进去。第二，也是更严重的——普通用户能不能通过修改 API 请求参数，把自己的 is_admin 从 false 改成 true？如果后端没有在权限中间件里保护这个字段，那就是一个提权漏洞。

这个案例的教训：权限/角色变更不是某一个领域的事，是数据库 + 后端权限 + 前端 UI 三方的事。Rulers 提供了跨域一致性检查的 checklist，但关键是——你要在 AI 加载规则时，确认三个领域都命中了。

下次做权限相关的变更，记住这个案例。

改规则不是直接打开文件就改——有标准流程。五步。

第一步，影响评估。在改之前，先读 RULER_MAINTENANCE.md，评估这次改动会影响哪些领域、哪些任务类型。不要低估一条规则的波及范围——特别是 core 目录下的规则。

第二步，修改规则文件。这步没太多要说的——就是编辑。

第三步，更新索引。如果新增或删除了文件，必须同步更新 INDEX.md。只是修改已有文件的内容，不需要动索引。

第四步，跑校验。这是安全网——改了任何东西都跑一下。

第五步，提交。遵循我们的 commit 规范：type 用英文（feat/fix/docs/chore），subject 用中文动宾结构。比如 "docs(backend): 更新 API 错误码规范，增加 4xx 系列定义"。

整个流程多花两分钟，但避免了改坏规则体系的风险。而且——规则变更应该和代码变更一样走 review 流程。PR 里包含校验通过截图，reviewer 确认没问题再合。

// part 05 · 答疑

常见问题速答

Q
必须用全部领域吗？

不用。没有移动端就删掉 frontend/app/，skill 初始化时会自动跳过不存在的领域。有什么平台生成什么规则。

Q
Level 0 能写代码吗？

不能。Level 0 只允许项目发现，必须先完成 PROJECT_PROFILE.md 审阅并升级到 Level 1。

Q
后来加了新平台怎么办？

重跑 skill。AI 自动检测新领域，补充生成对应规则。不会修改已有规则——增量更新。

Q
校验报错看不懂怎么办？

按错误提示逐项修。最常见：链接指向不存在文件 → 更新链接；INDEX 遗漏叶子文档 → 补充引用；metadata 缺失 → 补充必填字段。

快速过几个高频问题，练习中可能已经有同学遇到了。

"必须用全部领域吗？"——不用。你的项目有什么领域就生成什么规则。没有移动端，AI 就不会生成 frontend/app/ 的内容。这是"证据驱动"的体现。

"Level 0 能写代码吗？"——不能。这是硬约束。Level 0 AI 只看不动。你至少要完成 PROJECT_PROFILE.md 的审阅，升到 Level 1。

"后来加了新平台？"——重跑 skill。AI 会自动识别新领域，补充生成规则。已有的规则不受影响——增量更新，不是全部重来。

"校验报错看不懂？"——错误信息其实很直接。大部分是三种：链接指向不存在的文件（改了文件名但没更新引用）、INDEX 没覆盖新文件（加了文件但没更新索引）、metadata 字段缺失（新建规则文件时忘了写 metadata）。按错误提示一行一行修就行。

还有别的问题吗？现在可以提。

我们团队的Rulers 使用约定

05
新人入职一周内完成初始化

作为 onboarding checklist 的一部分。规则体系是最好的新人培训工具。

这是团队层面需要对齐的几件事——可能需要大家讨论一下，有些条款可以根据我们实际情况调整。

第一，规则目录纳入版本管理。documents/rulers/ 不是个人配置，是项目资产。所有人共享同一套规则——你对规则的修改会影响团队每个人的 AI 行为。所以改规则要谨慎、要走 review。

第二，单一来源原则。不确定放哪里时，先问。群里一句话的事，比后面清理冲突省心得多。

第三，规则变更走 review。PR 里贴校验通过截图——这是硬要求。规则 review 不需要像代码 review 那么细，但至少要有人确认"这条规则不会产生副作用"。

第四，月度评审。30 分钟不长的——快速过一下本月改了哪些规则、有没有覆盖盲区、有没有重复的规则该合并。

第五，新人 onboarding。新同事入职第一周内完成 Rulers 初始化——不是"有空再做"，是 onboarding checklist 的一部分。规则体系本身就是最好的培训工具——新人在用 AI 写代码的过程中，自然接触到团队规范。

这五条大家有什么意见？特别是月度评审的频率和时长——我们可以根据实际情况调整。

最后一句话送给大家——规范不是约束，是效率的放大器。

你前置花五分钟写的规则，会在后续每一次 AI 交互中帮你节省时间和认知负担。今天学了这么多，最关键的就是一件事：去你的项目里真的用起来。

打开终端，clone 仓库，安装 skill，对 AI 说出那句指令。几分钟后，你的项目就有了完整的 AI 开发护栏。之后每次用 AI 写代码，规则都在背后默默保护你——帮你统一风格、帮你拦截风险、帮你记住约定。

有问题随时找我或者提 issue。规则体系的完善是持续的过程——今天只是起点。谢谢大家！
